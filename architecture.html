Why this stack?

Neon solves the Supabase inactivity problem — data survives the NFL offseason without paying $25/mo. Next.js App Router gives us server components for fast initial loads (critical for draft night traffic spikes) and server actions for Dan's admin mutations. Drizzle keeps the ORM layer thin and type-safe. Auth.js starts as admin-only and scales to full user auth when pools launch. Everything auto-deploys from GitHub through the existing Vercel pipeline.

Schema notes

draft_boards supports both Dan's mock drafts and the actual draft results in the same structure. The type field distinguishes them. The scoring engine compares picks from a mock board against the actual board for the same season. Tables for pools, pool_members, and pool_scores will be added in Phase 3 when the social/competition features launch.

Build approach

Phase 1 is buildable in a focused sprint — it's essentially a CRUD app with a nice frontend. The schema is designed so Phase 2 (scoring) and Phase 3 (pools) bolt on without restructuring. The draft_boards + picks pattern works for Dan's mocks, user predictions, and actual results — the type field differentiates them. Drizzle migrations handle schema evolution as new tables arrive.
